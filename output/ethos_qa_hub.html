Area 1 — End-to-End Funnel & Document Testing
Verifies the complete customer journey from funnel entry through document delivery. Walk through the applicable product funnel, verify correct documents are generated, and confirm delivery via DocuSign and/or Member Portal. No insurance knowledge required — follow each step and mark Pass or Fail.
Pre-Test Setup
  • Obtain test applicant data (name, DOB, address, SSN) from test data sheet — do not use real SSNs
  • Confirm which form control is being tested (see Product Context card above)
  • Confirm test state for this scenario run
  • Confirm channel for this product (DTC, AA, Consumer, Partner as applicable)
  • Confirm replacement scenario: Yes or No
  • Have M&T log open to capture envelope IDs as you go
  • Confirm access to DocuSign test environment and Member Portal
Funnel Initiation
  • Start test application from Ethos funnel URL (test environment)
  • Verify product name and carrier displayed correctly in UI
  • Complete full interview funnel with test applicant data
  • Record all funnel answers for later field verification — especially all medical question answers
  • Verify correct form control is assigned to this application CRITICAL
DocuSign Envelope Creation & Routing
  • Verify DocuSign envelope is created after funnel completion
  • Verify envelope ID is captured in M&T log immediately
  • Verify AUTH packet appears FIRST in signing ceremony (before application) CRITICAL
  • Verify APPLICATION packet appears second (after auth)
  • Verify all required docs for the test state are present in the application packet
  • Verify signing ceremony can be completed end-to-end without errors
Underwriting & Policy Generation
  • After application submitted, verify underwriting decision is triggered
  • For straight-through (non-RUW): verify policy packet generated automatically
  • For RUW/IC: verify appropriate amendment or statement of good health is present per business rules
  • Verify POLICY packet is generated only after underwriting decision — not before
  • Verify policy packet delivered via DocuSign and/or Member Portal
  • Verify policy number populates correctly on contract pages
Delivery Verification
  • Verify all required packets delivered in correct order: Auth → App → Policy
  • Verify Member Portal shows correct policy status after signing
  • Verify Policy Delivery Receipt (if applicable to product) is present in policy packet
  • Verify applicant receives completion confirmation
  • Log envelope ID, scenario ID, and result in M&T log
  • If any failure: create Jira ticket before PROD promotion
Area 2 — Application Generation (Field Mappings, Checkboxes, Labels)
Verifies all fields on the insurance application form are correctly populated from the applicant's funnel responses. Every field on the printed/digital application must exactly match what the applicant answered.
⚠
CRITICAL: Never assert a hardcoded expected value for any field. Each field must be verified against the actual funnel response for the specific test applicant. Medical question answers must each be verified individually — mismatch is a contestability risk.
Proposed Insured Fields
Rule for ALL fields: Verify each field matches the funnel response for that specific test applicant. Do not assert a hardcoded expected value.
  • Name (First, Middle, Last) — verify against funnel response
  • Date of Birth (mm/dd/yyyy) — verify against funnel response
  • Social Security / Tax ID — verify against funnel response (test SSN only)
  • Current Mailing Address, City, State, Zip — verify each subfield against funnel response
  • Email Address — verify against funnel response
  • Cell Phone Number — verify against funnel response
  • Gender checkbox — verify correct option checked per funnel response
  • Marital Status checkbox — verify correct option checked per funnel response
  • Citizenship status — verify checkbox matches funnel response
  • Purpose of Insurance — verify checkbox matches funnel response
Beneficiary & Owner Fields
Rule: All beneficiary fields must match funnel response for beneficiary entry.
  • Beneficiary Name, DOB, Relationship — verify against funnel response
  • Benefit % — verify totals 100% across all beneficiaries per funnel response
  • Owner section — only present if owner ≠ insured per funnel response
  • Payor fields — verify match funnel response
Coverage & Financial Fields
Rule: Compute premium fresh — do not hardcode.
  • Coverage amount / face amount — verify against funnel response
  • Term / product type checkbox — verify correct option checked
  • Riders selected — verify against funnel response
  • Premium mode — verify against funnel payment selection
  • Annual income, employment status — verify against funnel response
Medical Questions CRITICAL — Contestability Risk
⚠
CRITICAL: Each medical question response must be verified INDIVIDUALLY against the test applicant's actual funnel answer. A checkbox mismatch on any medical question is a contestability risk. Do not batch-verify. Verify each question one at a time.
  • Each medical condition question — verify checkbox (Yes / None of the above / N/A) matches funnel response individually CRITICAL
  • Nicotine/tobacco status — verify matches funnel response; affects risk class CRITICAL
  • Height / Weight — verify against funnel response
  • Weight change — verify Yes/No and direction match funnel response
  • Family history conditions — verify each checkbox matches funnel response individually
  • Personal physician info — verify all subfields match funnel response
  • Any "Yes" responses — verify explanation text is populated in Additional Info section
Non-Medical & Signature Fields
Rule: City/State and Date on signature blocks must match actual test execution — not hardcoded.
  • Military / National Guard / Reserves question — verify checkbox matches funnel response (see Area 11 for military scenarios)
  • Replacement / existing insurance question — verify checkbox matches funnel response (see Area 7)
  • Criminal history, DUI, hazardous activities — verify each matches funnel response individually
  • Proposed Insured signature field — present and mapped in DocuSign
  • Proposed Insured City/State and Date — verify against signing location and date
  • Agent/producer signature field — present and mapped (if applicable)
Area 3 — Policy Document Generation
After underwriting approves an application, a policy packet is generated. This area verifies that the correct contract pages, schedule sections, and ancillary documents are present for each state and form control combination. Conditional schedule sections must be tested both ways (condition met / condition not met).
Policy Contract Pages
  • Verify correct state-specific contract pages are present for the test state
  • Verify policy contract form number matches PCS for the test state and form control
  • Verify policy number populated correctly on all contract pages
  • Verify insured name, DOB, face amount, coverage start date populated correctly
  • Verify premium amount computed correctly — do not hardcode expected premium
Schedule Sections (Conditional Logic)
Rule: Schedule sections with data-label conditions must be tested BOTH ways. Test the condition met (section PRESENT) and condition not met (section ABSENT).
  • Conditional schedule section: when year ≤ data label threshold — section PRESENT CRITICAL
  • Conditional schedule section: when year > data label threshold — section ABSENT CRITICAL
  • Log both results as separate test runs in M&T log
Ancillary Policy Documents
  • Policy Delivery Receipt — verify present in policy packet (if applicable to product)
  • Endorsements — verify state-specific endorsements present for test state
  • RUW Amendment / Statement of Good Health — verify present only when triggered by business rule
  • Rider disclosures — verify present only when corresponding rider is included on policy
M&T Evidence for Policy Docs
  • Screenshot of complete policy packet in DocuSign (all pages visible)
  • Screenshot confirming conditional schedule section logic (both states tested)
  • M&T log entry for each policy packet test run
Area 4 — Authorizations (Auth Packet)
The Auth packet contains carrier-agnostic HIPAA and authorization forms. These must be signed by the insured/applicant and appear FIRST in the DocuSign signing ceremony. This area verifies correct forms are present and correctly mapped.
Auth Packet — Required Forms
  • Auth packet appears FIRST in DocuSign signing ceremony CRITICAL
  • E-Sign Consent form — present and signed by insured
  • HIPAA Authorization — present and signed by insured; verify matches Legal Center version
  • Consumer Report Authorization — present and signed by insured
  • Carrier-specific HIPAA/Auth form — present if required by carrier for this product (see Area 6)
Auth Packet — Field Verification
Rule: Verify each auth form field matches funnel response for that applicant. Do not hardcode expected values.
  • Applicant name on HIPAA/auth forms — verify matches funnel response
  • Signature fields on auth forms — verify each is mapped to insured/applicant signer
  • Agent/producer signature on auth forms — verify mapping correct
  • State-specific auth forms — verify present for test state (see Area 8)
Area 5 — Options & Riders Coverage
This area tests rider and benefit option logic. Each rider has specific trigger conditions — the rider form must be present when triggered and absent when not triggered. Both states must be logged as separate test runs.
Rider Testing Framework (SOP Requirement)
Rule: Every rider and benefit option must be tested BOTH ways — condition triggered (form PRESENT) and not triggered (form ABSENT). Both results must be logged as separate test runs in M&T log.
  • Rider included — verify rider disclosure/form PRESENT in correct packet CRITICAL
  • Rider NOT included — verify rider disclosure/form ABSENT from packet CRITICAL
  • Verify rider form number matches PCS for the test state
  • Verify rider fields correctly populated from policy data
  • Log both triggered and not-triggered results as separate M&T log entries
Product-Specific Rider Checks
ⓘ
See the Forms Reference tab for the complete rider form list for the selected product. Use the business rule column to identify trigger conditions for each rider.
  • Accidental Death Benefit (ADB) rider — if applicable to product: verify disclosure present/absent per trigger
  • Accelerated Death Benefit riders (Care4Life, Terminal Illness, Chronic Illness) — verify each present/absent per trigger
  • Critical/Chronic/Terminal Illness riders — verify eligibility flags (ineligible_riders) correctly exclude forms
  • Perks Benefit — verify present/absent per state and trigger rules
Area 6 — Carrier HIPAA/Authorization Forms
Each carrier may have a carrier-specific HIPAA or authorization form in addition to the carrier-agnostic HIPAA. This area verifies the correct carrier-specific auth form is present for the product being tested.
Carrier-Specific Auth Form Checklist
  • Carrier HIPAA/auth form — verify correct carrier-specific form present for this product's carrier CRITICAL
  • Verify carrier auth form is in the Auth packet (not Application or Policy)
  • Verify carrier auth form fields correctly populated (applicant name, date, etc.) — verify against funnel response
  • Verify signature mapping: insured/applicant signs carrier auth form
  • Verify carrier auth form form number matches PCS for the test state and form control
Area 7 — Replacement Forms
When an applicant indicates they are replacing existing insurance, state-mandated replacement disclosure forms must be triggered. This area verifies correct trigger logic and form presence for each state. Both triggered and not-triggered states must be tested.
Replacement Form Testing (SOP Business Rule Requirement)
Rule: Test BOTH states for every replacement form. Both results must be logged as separate M&T log entries.
Condition Triggered replacing-existing-insurance = Yes (or terminate-existing-policy = Yes)
Replacement form MUST be PRESENT in packet. Verify form number matches PCS for the test state.
Condition Not Triggered replacing-existing-insurance = No
Replacement form MUST be ABSENT from packet. Verify no replacement form appears.
Replacement Trigger Logic
  • Ask Label replacing-existing-insurance = Yes — standard trigger for most states
  • Ask Label terminate-existing-policy = Yes — trigger for some compact states (see PCS Business Rule column)
  • AA channel: trigger logic may differ — verify coverage > 0 condition per PCS business rule
  • Verify state-specific replacement form number matches PCS for that state
Replacement Form Field Verification
Rule: Field values on replacement forms sourced from funnel — verify against actual funnel response, not hardcoded.
  • Existing policy details (company, policy number, face amount) — verify against funnel response
  • Applicant signature on replacement form — verify mapped in DocuSign
  • Agent signature on replacement form (if required) — verify mapped
  • Date on replacement form — verify against signing date
Area 8 — State-Specific Forms
Each state may require specific forms in addition to the standard packet. This area verifies that state-mandated forms are present for the test state and absent for states where they do not apply.
State-Specific Form Testing Framework
Rule: For each state-specific form: test the state where the form applies (form PRESENT) and test another state where it does not apply (form ABSENT). Log both as separate M&T log entries.
  • State-specific form tested in applicable state — form PRESENT CRITICAL
  • Same form tested in non-applicable state — form ABSENT CRITICAL
  • Verify state-specific form number matches PCS for that state
  • Verify form is in the correct packet (Auth, App, or Policy) per PCS doc assignment
  • Use Forms Reference tab to identify all state-specific forms for the selected product
Common State-Specific Scenarios to Test
  • CA — California-specific application, HIPAA, and any CA-only disclosures
  • FL — Florida-specific application and policy contract
  • SC — South Carolina-specific forms
  • VA — Virginia endorsement (if applicable to product)
  • TX — Texas-specific forms (if applicable)
  • Compact states vs non-compact states — verify correct form variant is used
Area 9 — Effective Dating & EFT
Policy effective dates and EFT (Electronic Funds Transfer) form dates must be computed correctly. This area verifies effective date logic and EFT form field mappings.
Effective Date Rule
Rule: Policy effective date must be computed fresh from actual test data — never hardcode. Verify the date satisfies: policy delivered + applicant accepted + first premium paid + no adverse health change.
  • Effective date on policy contract — compute from test data, verify formula conditions met CRITICAL
  • EFT form policy effective date — must match policy contract effective date
  • EFT form: first payment date — verify computed correctly from effective date
  • EFT bank routing number / account number — verify against funnel response for payment fields
  • EFT payment mode (monthly/quarterly/annual) — verify matches funnel payment selection
Area 10 — Business Rules & Conditionals
This area verifies all conditional business rules — where a form or section is included ONLY when specific conditions are met. Every conditional form must be tested both ways (triggered and not triggered) and both results logged in M&T.
Business Rule Testing Framework (SOP Requirement)
Rule: For every conditional form (Business Rule column non-empty in PCS): test the condition triggered (form PRESENT) AND condition not triggered (form ABSENT). Both results must be logged as separate M&T log entries.
Condition Triggered Business rule condition = TRUE
Form MUST be PRESENT in correct packet. Log as PASS if present, FAIL if absent.
Condition Not Triggered Business rule condition = FALSE
Form MUST be ABSENT from packet. Log as PASS if absent, FAIL if present.
Business Rules — From Forms Reference
ⓘ
Open the Forms Reference tab and filter by the selected product. Rows with a yellow left border have active business rules. Test each one both ways.
  • Replacement trigger (replacing-existing-insurance / terminate-existing-policy) — test Yes and No
  • Military status trigger (military-status-current = Yes) — test Yes and No
  • RUW vs IC application path — test both paths where applicable
  • Rider eligibility flags (ineligible_riders values) — test eligible and ineligible
  • Age-based triggers (e.g. age ≥65 CA disclosure) — test age ≥65 and age <65
  • Face amount triggers (small face amount disclosures) — test face ≤ threshold and face > threshold
  • Schedule section data label conditions — test year ≤ label and year > label
  • Reissue/non-reissue (Statement of Good Health for Banner SIWL) — test both
Area 11 — Military Forms
For applicants who are current military / National Guard / Reserves, a military-specific disclosure form is required. Both triggered and not-triggered states must be tested.
Military Form Testing
Rule: Test BOTH states. Both must be logged as separate M&T log entries.
Military = Yes (Triggered) Ask Label: military-status-current = Yes
Military disclosure form MUST be PRESENT. Verify in correct packet.
Military = No (Not Triggered) Ask Label: military-status-current = No
Military disclosure form MUST be ABSENT from packet.
  • Military form present when military-status-current = Yes CRITICAL
  • Military form absent when military-status-current = No CRITICAL
  • Verify military form number matches PCS for the product and state
  • Verify military form is in the correct packet per PCS doc assignment
  • Verify military form fields populated correctly from funnel response
  • Log both results as separate M&T log entries with different scenario IDs
Products Without Separate Military Form
ⓘ
Some products (e.g. LGA Prime) do not have a separate military form — instead the military question appears within the application itself (e.g. Section X Q4). In this case, verify the checkbox on the application matches the funnel response — not a separate form. See the Forms Reference tab for the selected product.
Area 12 — E-Sign Scenarios
This area verifies all digital signature flows. Every signature field must be correctly mapped to the right signer in DocuSign, and the signing ceremony must complete without errors for all applicable signers.
E-Sign Testing Checklist
  • Auth packet: all auth/HIPAA forms — insured/applicant signature field present and mapped CRITICAL
  • Application: insured signature on application — correctly mapped to insured signer
  • Application: agent/producer signature (if required) — correctly mapped to agent signer
  • Policy packet: insured signature on policy delivery receipt — correctly mapped
  • All signers complete their signing steps without DocuSign errors
  • Signing ceremony order: Auth first → App second → Policy third
  • Confirmation emails received by all signers after completion
  • Signed documents available in DocuSign completed envelope and Member Portal
E-Sign Edge Cases
  • Owner ≠ Insured: verify owner has separate signature step where applicable
  • RUW/IC activation: verify amendment/statement of good health signature step works
  • Multiple signers: verify each signer sees only their required forms
  • Declined to sign: verify error handling and M&T log entry
E-Sign M&T Evidence
  • Screenshot of DocuSign envelope showing all signer routing tabs
  • Screenshot of completed signing ceremony for each signer
  • Screenshot confirming all documents delivered after completion
  • DocuSign Envelope ID recorded in M&T log for each test run
Area 13 — Buyer's Guide
A Buyer's Guide is a carrier-agnostic document that must be appended to the correct packet. This area verifies the correct Buyer's Guide version is present for each state and product.
Buyer's Guide Testing Checklist
  • Buyer's Guide present in correct packet — verify against PCS doc assignment for this product CRITICAL
  • Verify correct Buyer's Guide form number for the test state (standard vs state-specific variant)
  • Buyer's Guide is NOT signed (it is an informational document — not a signature form)
  • Buyer's Guide appended correctly — verify page order in packet
  • State-specific Buyer's Guide (e.g. ME, NJ for LGA Prime) — verify correct variant used
